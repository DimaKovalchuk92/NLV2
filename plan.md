# E02: Workspace Personalization - План Реализации

## 1. Архитектурные Решения (MVP Focus)

Задача будет реализована полностью на стороне клиента, используя существующую архитектуру (`MockAPI`, `localStorage`).

1.  **Централизованная функция:** В `admin.html` будет создана единая JavaScript-функция `loadSpecialistProfile()`.
2.  **Получение данных:** Эта функция будет вызвана при загрузке страницы. Она обратится к `MockAPI.getLoggedInUser()` для получения объекта текущего пользователя, включая вложенный объект `profile`.
3.  **Обновление хедера:** Функция найдет по `id` элементы в шапке (`#specialist-name`, `#specialist-spec`, `#specialist-avatar`) и обновит их содержимое (`textContent`) данными из `user.profile`.
4.  **Обновление настроек:** Функция также найдет по `id` новые элементы в модальном окне настроек (`#admin-settings-modal`) и заполнит их данными из профиля (`fullName`, `specializations`, `bio`, `welcomeMessage`).

## 2. Изменения в `admin.html`

### HTML:
-   **Header:** Элементам в шапке будут присвоены `id` для легкого доступа из JavaScript.
-   **Модальное окно настроек:** Внутрь `#admin-settings-modal` будет добавлен новый `div`-контейнер для раздела "Мой профиль". Внутри него будут размещены параграфы `<p>` или `<div>` с `id` для каждого поля профиля (например, `#profile-display-name`, `#profile-display-spec` и т.д.).

### JavaScript:
-   **Создание функции:** Будет написана функция `loadSpecialistProfile()`.
-   **Вызов функции:** `loadSpecialistProfile()` будет добавлена в обработчик `DOMContentLoaded` для выполнения при загрузке страницы.
-   **Открытие настроек:** Функция `openAdminSettingsModal()` будет дополнена вызовом `loadSpecialistProfile()`, чтобы гарантировать, что данные в модальном окне всегда актуальны при его открытии.

## 3. Изменения в `modules/mock-api.js`

-   На данном этапе изменения в `MockAPI` не требуются, так как все необходимые данные уже сохраняются в `localStorage` в рамках Эпика E01. Мы будем использовать только функцию `getLoggedInUser()`.
