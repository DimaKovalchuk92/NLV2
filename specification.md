# E02: Workspace Personalization - Спецификация

## 1. Контекст

После того как специалист успешно прошел онбординг (Эпик E01), его рабочее пространство (`admin.html`) должно стать персонализированным. Отображение реальных данных профиля вместо статичного текста подтверждает, что информация сохранена, и создает более профессиональную и доверительную среду для работы.

## 2. Цели (Goals)

1.  **Синхронизировать данные профиля с интерфейсом:** Заменить статичные заглушки в кабинете специалиста на реальные данные (ФИО, специализация), полученные из `localStorage` через `MockAPI`.
2.  **Создать точку доступа к профилю:** Добавить в интерфейс блок или ссылку "Мой профиль", где специалист сможет просмотреть всю введенную при регистрации информацию.

## 3. Критерии Принятия (Acceptance Criteria)

-   **AC1:** При загрузке `admin.html` в шапке (`header`) вместо "Панель Специалиста" отображается полное имя (`fullName`) из профиля пользователя.
-   **AC2:** Под именем в шапке отображается основная специализация (первый элемент из массива `specializations`).
-   **AC3:** В модальном окне "Настройки" создан новый раздел "Мой профиль", где в текстовом виде (только для чтения) отображаются: ФИО, специализация, биография и приветственное сообщение.

## 4. Не-цели (Non-Goals)

-   **Редактирование данных:** На данном этапе мы **НЕ** реализуем возможность изменять данные профиля. Все поля предназначены только для отображения.
-   **Сложная верстка:** Блок "Мой профиль" реализуется в максимально простом виде внутри существующих компонентов (например, модального окна настроек).
-   **Интерактивность:** Приветственное сообщение просто отображается в настройках и пока никуда не отправляется.
